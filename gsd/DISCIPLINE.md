# GSD Discipline Rules

Золотые правила дисциплинированной разработки.

## 🚫 Запреты (Never)

### 1. Нет коду без failing-теста

```
❌ НЕ писать код до написания теста
✅ СНАЧАЛА красный тест (RED), потом код (GREEN)

Правило:
- Тест должен FAIL до реализации
- Тест покрывает happy path
- Тест покрывает edge cases
- Тест покрывает error cases
```

**Проверка:**
```bash
# Перед коммитом
npm test  # Должен быть PASS после реализации
```

---

### 2. Нет мерджу без code review

```
❌ НЕ merge в main без review
✅ ВСЕГДА запускать skill:review перед merge

Правило:
- Correctness проверена
- Security проверена
- Code quality проверена
- Performance проверена
```

**Проверка:**
```
gsd:ship --dry-run  # Пре-проверка перед ship
```

---

### 3. Нет "готово" без верификации

```
❌ НЕ声称 "исправлено" или "готово" без доказательств
✅ ВСЕГДА запускать verification checklist

Правило:
- Все acceptance criteria проверены
- Все тесты PASS (100%)
- Покрытие ≥ 80%
- Lint проходит без ошибок
```

**Проверка:**
```bash
gsd:verify-work N
```

---

### 4. Нет работы без задачи

```
❌ НЕ начинать работу без задокументированных требований
✅ СНАЧАЛА gsd:requirements-spec, потом код

Правило:
- Требования имеют ID (REQ-01...)
- Acceptance criteria определены
- Тест-кейсы подготовлены
```

**Проверка:**
```
gsd:requirements-spec N  # Перед plan-phase
```

---

## ✅ Обязательства (Always)

### 1. Evidence Before Claims

```
✅ ВСЕГДА предоставлять доказательства:
- Скриншоты тестов (PASS)
- Логи верификации
- Отчёты code review
- Метрики покрытия
```

**Пример:**
```
✓ Исправлено

Доказательства:
- Tests: 15/15 passed
- Coverage: 87%
- Lint: no errors
- Review: approved
```

---

### 2. TDD Cycle (RED-GREEN-REFACTOR)

```
✅ ВСЕГДА следовать циклу:

1. RED    → Написать failing тест
2. GREEN  → Написать код для PASS
3. REFACTOR → Улучшить код (тесты PASS)
4. COMMIT → Закоммитить изменения
```

**Чек-лист:**
- [ ] Тест написан ДО кода
- [ ] Первый запуск: FAIL
- [ ] Код минимален для PASS
- [ ] Рефакторинг выполнен
- [ ] Тесты закоммичены с кодом

---

### 3. Dependency Graph Before Execute

```
✅ ВСЕГДА проверять граф зависимостей перед выполнением

Правило:
- Определять ready планы (без блокировок)
- Находить критический путь
- Избегать bottlenecks
```

**Проверка:**
```
gsd:dependency-graph N
gsd:ready
```

---

### 4. Isolate Features

```
✅ ВСЕГДА использовать git worktrees для фич

Правило:
- Одна фича = один worktree
- Main worktree только для review/merge
- Удалять worktree после merge
```

**Проверка:**
```bash
git worktree list  # Проверка активных worktrees
```

## 📋 Чек-листы по Этапам

### Перед Началом (Pre-Implementation)

- [ ] Требования задокументированы (gsd:requirements-spec)
- [ ] Компромиссы исследованы (gsd:discuss-phase --analyze)
- [ ] Планы созданы (gsd:plan-phase)
- [ ] Граф зависимостей построен (gsd:dependency-graph)
- [ ] Worktree создан (gsd:using-git-worktrees)

### Перед Коммитом (Pre-Commit)

- [ ] Все тесты PASS
- [ ] Покрытие ≥ 80%
- [ ] Lint без ошибок
- [ ] Коммит message описательный

### Перед Merge (Pre-Merge)

- [ ] Code review выполнено (skill:review)
- [ ] Verification checklist пройден (gsd:verify-work)
- [ ] Все acceptance criteria проверены
- [ ] Нет unresolved comments от review

### Перед Ship (Pre-Ship)

- [ ] UAT пройден
- [ ] Документация обновлена
- [ ] Архив создан (gsd:compact, если фаза завершена)
- [ ] STATE.md обновлён

---

## Нарушения и Последствия

| Нарушение | Последствие | Исправление |
|-----------|-------------|-------------|
| Код без теста | Отклонить PR | Написать тесты, запустить TDD |
| Merge без review | Откатить merge | Провести review, re-merge |
| "Готово" без верификации | Перепроверить | Запустить gsd:verify-work |
| Работа без задачи | Остановить работу | Создать requirements-spec |

---

## Enforcement

**Автоматические проверки:**

```
Перед gsd:execute-phase:
→ Проверка: requirements-spec выполнен?
→ Проверка: dependency-graph построен?

Перед gsd:ship:
→ Проверка: skill:review запущен?
→ Проверка: gsd:verify-work пройден?

Перед gsd:compact:
→ Проверка: все тесты PASS?
→ Проверка: документация обновлена?
```

---

## Культура

**Мы верим в:**

- 📊 **Данные вместо мнений** — метрики, тесты, логи
- 🔍 **Прозрачность** — видимый прогресс, открытые issues
- 📈 **Непрерывное улучшение** — ретроспективы, lessons learned
- 🤝 **Взаимное ревью** — конструктивная критика, обучение

**Мы не терпим:**

- ❌ "Работает на моей машине"
- ❌ "Потом напишу тесты"
- ❌ "И так сойдёт"
- ❌ "Не моя задача"
