# gsd:requirements-spec

Уточнение требований перед планированием фазы.

## Что делает

1. Анализирует требования из PROJECT.md/REQUIREMENTS.md
2. **Задаёт уточняющие вопросы** для неясных моментов
3. Создаёт детальную спецификацию требований
4. Определяет acceptance criteria
5. Формирует traceability matrix

## Как использовать

Напишите мне:
```
gsd:requirements-spec 1
```
или для обсуждения:
```
gsd:discuss-phase 1 --analyze
```

## Процесс

### Шаг 1: Анализ требований

**Читаю существующие требования:**
- PROJECT.md — основные требования
- REQUIREMENTS.md — детальные требования с ID
- Предыдущие фазы — зависимости

**Классифицирую требования:**
```
### Validated (подтверждены)
- REQ-01: [требование] — критичность: High

### Active (в работе)
- REQ-02: [требование] — критичность: Medium

### Out of Scope (не в фазе)
- REQ-03: [требование] — отложено
```

### Шаг 2: Уточняющие вопросы

**Если требования неясны, задаю вопросы:**

| Категория | Вопросы |
|-----------|---------|
| **Функциональность** | Что именно должно происходить? Какие входные/выходные данные? |
| **Пользователи** | Кто использует? Какие сценарии? |
| **Ограничения** | Производительность, безопасность, совместимость? |
| **Edge cases** | Что при пустых данных? Ошибках сети? |
| **Зависимости** | От чего зависит? Что зависит от этого? |

**Формат вопросов:**
```
## Уточнения для Phase 1

### Вопрос 1: [тема]
- Контекст: [почему спрашиваю]
- Варианты:
  A) [вариант 1] — плюсы/минусы
  B) [вариант 2] — плюсы/минусы
- Рекомендация: [мой совет]
```

### Шаг 3: Спецификация требований

**Создаю/обновляю REQUIREMENTS.md:**

```markdown
# Requirements Specification

## Phase 1: [Название]

### REQ-01: [Название требования]
**Описание**: [детальное описание]
**Приоритет**: High/Medium/Low
**Acceptance Criteria**:
- [ ] Критерий 1
- [ ] Критерий 2
- [ ] Критерий 3

**Test Cases**:
- TC-01-01: Happy path — [описание]
- TC-01-02: Edge case — [описание]
- TC-01-03: Error case — [описание]

**Dependencies**: [от чего зависит]
**Status**: Validated / Active / Out of Scope
```

### Шаг 4: Traceability Matrix

**Связываю требования с планами:**

```
| Требование | Планы | Тесты | Статус |
|------------|-------|-------|--------|
| REQ-01     | 01-01, 01-02 | TC-01-01..03 | Validated |
| REQ-02     | 01-03 | TC-02-01..02 | Active |
```

### Шаг 5: Verification

**Проверяю полноту:**
- [ ] Все требования имеют ID
- [ ] Все требования имеют acceptance criteria
- [ ] Все требования имеют тест-кейсы
- [ ] Все требования приоритизированы
- [ ] Все неясности разрешены
- [ ] Traceability matrix заполнен

## Результат

```markdown
## Requirements Complete ✓

**Phase 1**: [Название]
**Требований**: 12
- Validated: 8
- Active: 3
- Out of Scope: 1

**Acceptance Criteria**: 35
**Test Cases**: 48

Готово к планированию: `gsd:plan-phase 1`
```

## После завершения

Следующая команда: `gsd:plan-phase 1`

## Связь с другими воркфлоу

- **До**: `gsd:new-project` или `gsd:discuss-phase N`
- **После**: `gsd:plan-phase N`
- **TDD**: Тест-кейсы из spec используются в `gsd:execute-phase N`
